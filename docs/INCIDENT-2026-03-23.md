# Incident 2026-03-23

## 标题

`daily-reminder_VPS` 在部分 OpenClaw 环境中存在“提醒判断成功但未实际投递到 Feishu”的链路缺口，同时午夜清理 prompt 可能泄露占位词。

## 背景

这份记录整理自一次跨机器实跑后的问题反馈，以及随后对仓库默认实现的修订。

## 现象

- 用户已成功通过 `每日提醒 开始 ...` 录入当天任务
- `state.json` 正常生成
- checker cron 运行成功
- `build-reminder` 能正确判断 `periodic / special / merged / catchup`
- 但飞书未收到应有提醒

同时，另一台机器上还出现了午夜收到“静默”两个字的异常现象。

## 确认过的根因

### 根因 1

早期安装器只改写 `~/.openclaw/cron/jobs.json`，但 Gateway scheduler 不会自动热加载该文件。

结果：

- 文件里有 job
- scheduler 里没有 job
- 提醒根本不会被调度

### 根因 2

即使 scheduler 已加载，旧版 checker 默认结构仍是：

- `sessionTarget = "main"`
- `payload.kind = "systemEvent"`

这要求 OpenClaw 主会话再完成一次：

1. 收到 system event
2. 调用 `build-reminder`
3. 生成提醒文本
4. 再把消息投递到飞书

在部分环境里，这条链路并没有闭环，表现为：

- `lastRunStatus = ok`
- `lastDeliveryStatus = not-requested`

也就是任务判断发生了，但真正的消息发送没有被请求。

### 根因 3

午夜清理 prompt 使用了“静默执行”之类表述，在某些实现路径下，这种占位词会被错误输出并发送给用户。

## 修复动作

### 已做的修复

- 安装器继续保留 live scheduler sync，避免只有文件更新、没有内存注册
- checker 默认改为 `isolated + agentTurn + announce`
- midnight clear 默认改为 `isolated + agentTurn + no-deliver`
- `install_cron.py` 支持 `--channel` / `--to` / `--account`
- `install.sh` 支持通过环境变量透传上述投递参数
- midnight clear prompt 移除“静默”这类占位词

### 修复后的设计

checker job 现在直接完成：

1. 运行 `build-reminder`
2. `kind == none` 时输出 `HEARTBEAT_OK`
3. `kind != none` 时只输出 `message`
4. 由 cron `delivery.mode = "announce"` 直接发到飞书

midnight clear job 现在直接完成：

1. 运行 `clear-day`
2. 清空当天状态
3. 因为 `delivery.mode = "none"`，不会投递任何消息

- clear prompt 中不再包含“静默”

## 对目标机的建议

部署到新机器或多账号环境时，建议显式指定：

- `DAILY_REMINDER_CHANNEL=feishu`
- `DAILY_REMINDER_TO=user:YOUR_TARGET`
- `DAILY_REMINDER_ACCOUNT=main`

这样可以避免依赖 last-route，进一步降低“判断正常但未投递”的概率。
